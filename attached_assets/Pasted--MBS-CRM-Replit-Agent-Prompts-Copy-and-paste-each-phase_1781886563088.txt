Build a full-stack CRM application for a business financing brokerage called My Business Solutions (MBS). Use PostgreSQL with Drizzle ORM, React with TypeScript for the frontend, and Express for the API server. Set up Replit Auth or Clerk for authentication.

DATABASE SCHEMA — Create these tables:

users: id, name, email, role (enum: admin/manager/rep), created_at

activity_log: id, user_id (FK), action (string), entity_type (string), entity_id (string), details (jsonb), created_at

Add indexes on leads.email, leads.phone, leads.ein, leads.status, leads.assigned_rep_id.

- Admin: full access to everything
- Manager: read/write all leads, can reassign, can view team metrics

NEW API ENDPOINTS:

POST /api/applications/submit — PUBLIC endpoint, rate-limited. Accepts multipart form data with all application fields + bank statement PDF uploads. Processing steps:
  1. Validate all required fields
  2. Run duplicate check (email, phone, EIN)
  3. Create the lead with status "application_received" and source "website"
  4. Store the application record with encrypted SSN
  5. Upload bank statement PDFs to Object Storage, save to documents table
  6. For each bank statement PDF, trigger OCR extraction (see below)
  7. Auto-assign to a rep (round-robin among active reps)
  8. Log everything to activity_log
  9. Return { success: true, lead_id }

BANK STATEMENT OCR — When a bank statement PDF is uploaded, process it server-side:

1. Read the PDF content (use pdf-parse or similar to extract text, or send as image to Claude)
2. Send to the Claude API (ANTHROPIC_API_KEY from Replit Secrets) with this system prompt:

"You are a bank statement parser. Extract the following from this bank statement and return ONLY valid JSON, no other text:
{
  "statement_month": number (1-12),
  "statement_year": number,
  "total_deposits": number,
  "average_daily_balance": number,
  "nsf_count": number (count of NSF/returned items),
  "negative_balance_days": number,
  "existing_positions": [
    { "amount": number, "frequency": "daily" or "weekly", "description": "string" }
  ]
}
For existing_positions, look for recurring debits that appear daily or weekly with consistent amounts — these indicate existing merchant cash advance or loan payments. If none found, return an empty array."

3. Parse the JSON response and store in bank_statement_extractions
4. After all statements are processed, calculate aggregates and store on the lead record

PHASE 4 — LENDER MATCHING ENGINE
================================================================================

Build an automated lender-matching engine that evaluates leads against configurable lender criteria.

NEW TABLES:

lenders: id, name, program_types (text[] — array of 'equipment' and/or 'working_capital'), min_amount (integer), max_amount (integer), min_credit_score (integer, nullable), accepted_industries (text[] — array), min_time_in_business_months (integer), accepted_states (text[] — array, empty = all states), max_existing_positions (integer, default 10), priority_weight (integer, 1-10, default 5), contact_name, contact_email, notes, is_active (boolean, default true), created_at, updated_at

lender_matches: id, lead_id (FK), lender_id (FK), match_score (integer, 0-100), criteria_breakdown (jsonb — array of { criterion, passed, detail }), matched_at

lender_submissions: id, lead_id (FK), lender_id (FK), submitted_by (FK users), status (enum: submitted/pending/approved/declined/withdrawn), response_notes, submitted_at, updated_at

MATCHING ALGORITHM — Create a function matchLeadToLenders(leadId) that:

1. Load the lead with its application data and financial data (from Phase 2 OCR if available)
2. Load all active lenders
3. For each lender, evaluate these criteria:
   - Application type matches lender's program_types → pass/fail
   - Requested amount between lender's min_amount and max_amount → pass/fail with detail
   - Credit score >= lender's min_credit_score (skip if no score yet) → pass/fail
   - Lead's industry in lender's accepted_industries (skip if empty array = all) → pass/fail
   - Lead's time_in_business >= lender's min_time_in_business_months → pass/fail
   - Lead's state in lender's accepted_states (skip if empty = all) → pass/fail
   - Detected existing positions <= lender's max_existing_positions (use OCR data if available) → pass/fail
4. Calculate score: (passed_criteria / total_applicable_criteria) * 100, weighted by priority_weight
5. Return ranked list sorted by score descending, each with the full criteria breakdown
6. Store results in lender_matches

AUTO-TRIGGER: When a lead's status changes to "application_received", automatically run the matching algorithm. Store results. Log to activity_log.

API ENDPOINTS:

GET /api/lenders — list all lenders (admin/manager only)
POST /api/lenders — create a lender (admin only)
PUT /api/lenders/:id — update a lender (admin only)
DELETE /api/lenders/:id — soft delete (set is_active = false)

POST /api/leads/:id/match — manually trigger matching for a lead. Returns the ranked results.
GET /api/leads/:id/matches — get stored match results for a lead

POST /api/leads/:id/submissions — mark a lead as submitted to a lender. Body: { lender_id }. Creates a lender_submissions record. Logs to activity_log.
GET /api/leads/:id/submissions — list all submissions for a lead
PUT /api/submissions/:id — update submission status (approved/declined/etc.)

FRONTEND:

- "Submit to Lender" button per lender → confirmation modal → creates submission record

DRIP ENGINE — Create a background job that runs every 10 minutes (use setInterval or a cron library):
1. Query drip_enrollments where status = 'active'
2. For each enrollment, get the next step (current_step + 1)
3. Check if enough time has elapsed since last_step_sent_at (or enrolled_at for step 1) based on the step's delay_hours
4. If due: send the email using the step's template, increment current_step, update last_step_sent_at
5. If current_step > total steps: set status to 'completed', set completed_at

PHASE 9 — CREDIT INTEGRATION (EXPERIAN)
================================================================================

Integrate Experian credit pulls into the CRM with secure storage and compliance logging. Requires EXPERIAN_API_KEY, EXPERIAN_API_SECRET, and EXPERIAN_API_URL in Replit Secrets. The specific API endpoints and authentication flow will depend on the Experian product contracted — adjust accordingly.

NEW TABLES:

credit_pulls: id, lead_id (FK), pulled_by (FK users), pull_type (enum: soft/hard), consent_captured_at (timestamp), consent_ip (string), credit_score (integer, nullable), report_summary (jsonb — parsed key data), request_payload_encrypted (text — AES-256), response_payload_encrypted (text — AES-256), status (enum: pending/completed/error), error_message (nullable), created_at

credit_compliance_log: id, credit_pull_id (FK), lead_id (FK), user_id (FK), action (string), permissible_purpose (string), details (jsonb), created_at
— THIS TABLE IS APPEND-ONLY. Do NOT create any UPDATE or DELETE routes for it. At the database level, revoke UPDATE and DELETE permissions on this table if possible.

ENCRYPTION: Use the same AES-256-GCM encryption utilities from Phase 2 (ENCRYPTION_KEY secret). Encrypt the full Experian request and response payloads before storing. The parsed report_summary in credit_pulls contains only the data needed for the UI — never raw SSNs or full account numbers.

API ENDPOINTS:

POST /api/leads/:id/credit/consent — capture consent before any pull. Body: { consent_type: "credit_pull", agreed: true }. Store consent_captured_at and consent_ip on the lead record. Return { consent_id }.

POST /api/leads/:id/credit/pull — initiate a credit pull. Steps:
  1. Verify consent exists and is recent (within last 30 days). Reject if not.
  2. Get the lead's application data (owner name, SSN [decrypted server-side], DOB, address)
  3. Call the Experian API with the required parameters
  4. Encrypt the full request and response payloads
  5. Parse the response to extract: credit_score, tradeline_summary (array of { creditor, balance, status, payment_history }), inquiry_count, derogatory_count, public_records_count
  6. Store everything in credit_pulls
  7. Write to credit_compliance_log: { action: "credit_pull", permissible_purpose: "credit application evaluation", details: { score, pull_type } }
  8. Log to activity_log (WITHOUT any PII — just "Credit report pulled, score: XXX")
  9. Return the parsed summary (never the raw response)

GET /api/leads/:id/credit — get all credit pull results for a lead (most recent first). Returns the parsed summaries, not raw data.

GET /api/credit/compliance-log — admin only. Paginated list of all credit pulls across all leads. Filterable by date range, rep, lead. For audit and compliance review.
GET /api/credit/compliance-log/export — CSV export of the compliance log.
