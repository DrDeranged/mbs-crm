TASK 1 — SENTRY ERROR TRACKING (server + frontend, PII-SAFE)
================================================================================
Install: pnpm --filter @workspace/api-server add @sentry/node
         pnpm --filter @workspace/mbs-crm add @sentry/react
Do NOT install @sentry/profiling-node or enable any profiling/tracing — native binaries
do not compile on this platform, and we only want error monitoring.

SERVER (artifacts/api-server):
- Initialize Sentry as early as possible in the entry point, ONLY if process.env.SENTRY_DSN
  is set (if absent, skip init entirely and log one info line — the app must run
  identically without it).
- CRITICAL PII CONFIGURATION — the application-submit endpoint's request body contains
  SSNs. Sentry must NEVER receive request bodies or PII:
    Sentry.init({
      dsn: process.env.SENTRY_DSN,
      sendDefaultPii: false,
      maxRequestBodySize: "none",   // or the current option name for disabling body capture
      beforeSend(event) {
        // strip any request data defensively
        if (event.request) { delete event.request.data; delete event.request.cookies; delete event.request.headers; }
        return event;
      },
    });
  Verify against the installed SDK version's option names; the requirement is: no request
  bodies, no cookies, no headers, no user PII — stack trace, message, URL path, and our
  request id only.
- Integrate with the EXISTING global error handler in app.ts: on 500-level errors, call
  Sentry.captureException(err) with the request id attached as a tag
  (Sentry.setTag / scope tag "request_id"). Keep the existing error_log table write —
  Sentry is in ADDITION to, not instead of, our internal logging.
- Also capture errors from the background jobs: in the catch blocks of dripJob,
  taskReminderJob, renewalJob (and the new backup job from Task 2), call
  Sentry.captureException(err) guarded by the same DSN-present check.

FRONTEND (artifacts/mbs-crm):
- Initialize @sentry/react in the entry (main.tsx) ONLY if import.meta.env.VITE_SENTRY_DSN
  is present. Error monitoring only: no tracing, no replay, no profiling.
    Sentry.init({ dsn: import.meta.env.VITE_SENTRY_DSN, sendDefaultPii: false });

- Create artifacts/api-server/src/lib/backupJob.ts exporting runBackupJob():
  - Overlap lock (running flag) exactly like the other jobs.

  - Errors: captureException to Sentry (if configured) + job_runs error status; never
